A Standards-Based Explorer for Environmental Data

Finding, understanding, and visualizing data through its metadata

  • Search a data catalogue by where, when, and what — then view the data in the browser.
  • Built entirely on open standards, not custom formats.
Say: A small tool with a big idea — if data is described well, software can find it and show it automatically, with no special handling per dataset.
Motivation

We have lots of data. Using it is the hard part.

  • Environmental data lives in many archives, formats, and projections.
  • Users don't want files — they want to find the right thing and look at it.
  • The gap between "data exists" and "data is usable" is metadata.
Say: The bottleneck isn't storage — it's discovery and interpretation. Metadata is the bridge, and this project is a test of how well that bridge holds.
What it does

Search → understand → visualize, all from the catalogue

  • Search by area (draw a box on a map), time range, and free text.
  • The tool reads each result's metadata to decide what it is and how to show it.
  • Visualize: plot a time series or profile inline, or overlay a map layer.
Say: The user never touches a file path or a format. They draw a box, pick a date, and the catalogue's metadata does the rest.

The key idea

The metadata decides the experience

  • CF "featureType" (time series, profile, trajectory…) tells the tool how to plot the data — automatically.
  • Service links in the record (WMS, OPeNDAP) tell it how to access the data.
  • Result: no per-dataset programming. Good metadata = instant usability.
Say: A field called "featureType" literally chooses the right chart. When metadata is rich, the tool just works. When it isn't, the tool can't help — which is a powerful diagnostic.
Strengths

What's genuinely good

  • Interoperable — open standards, no vendor or format lock-in.
  • Metadata-driven — behavior comes from the catalogue, not hard-coded.
  • No data duplication — reads existing archives in place.
  • Polar-ready — supports Arctic projections (Svalbard, UPS North) most web maps ignore.
  • Lightweight — runs in a browser, already live in our infrastructure.
Weaknesses & hard lessons

Honest limitations — most are about data, not code

  • Only as good as the metadata. Missing featureType or service links = invisible to the tool.
  • Catalogue noise. Millions of fine-grained satellite records make useful datasets sparse and searches heavy.
  • Inconsistent catalogues. One returned different results for the same query — a server-side compliance gap.
  • Projection fragmentation. Services advertise different coordinate systems; polar data needs care.
  • Performance depends on remote servers, not on us.
Say: The most interesting finding: most of our problems were metadata-quality problems, not software problems. The tool is a mirror held up to our data management.
Response

Turning weaknesses into guardrails

  • Guard against runaway queries — require a filter; warn when a search is too broad.
  • Search results several times faster by querying the catalogue in parallel.
  • Bundled our own Norwegian + Svalbard coastline where no standard polar basemap existed.
  • A layer manager to control map overlays (order, transparency, visibility).
Say: Each limitation pushed a concrete improvement — several are reusable building blocks for other tools.
Status

Where we are today

  • 🟢 Usable working prototype, deployed in our operational environment.
  • 🟢 Refactored into reusable, shared components (search, plotting, maps).
  • 🟢 Automated test suite (130+ tests) for confidence in changes.
  • 🟡 Validated mainly against met.no catalogues — broader testing pending.
Roadmap

Further development

  • Collection-level search (datasets, not millions of granules) for speed and clarity.
  • Caching & smarter ranking of the most relevant results.
  • Richer visualization: vertical profiles, time animation of map layers, downloads.
  • A metadata feedback loop — flag records missing featureType/links to improve the catalogue itself.
  • Wider reach: point it at other national/European catalogues.
Why it matters

A practical test of FAIR data

  • The tool only works because data is Findable, Accessible, Interoperable, Reusable.
  • Where it struggles, it points directly at metadata we can improve.
  • Open standards + good metadata = data that serves people, not just storage.
Say: If you take one thing away — investing in metadata quality and open standards pays off directly in tools like this. The software is the easy part; the metadata is the asset.
